# 交互模型与技术架构

## 一、交互模型

### 1.1 核心原则：越轻越好

所有操作通过飞书机器人对话完成，不需要打开额外 App。未来可扩展语音交互。

### 1.2 交互场景矩阵

| 场景 | 触发方 | 交互方式 | 示例 |
|------|--------|---------|------|
| 创建 TDL | 用户 | 对话/语音 → 机器人 | 「下周三前审核斯坦星球暑期班课程方案」 |
| 查询 TDL | 用户 | 对话 → 机器人 | 「我今天有哪些待办」「励步这周还有什么没做完」 |
| 更新 TDL | 用户 | 对话 → 机器人 | 「把斯坦星球那条改到周五」「标记诺伊曼那条已完成」 |
| 催收提醒 | 系统 | 机器人主动推送 | 到期/逾期/周期汇总自动发消息 |
| 反馈报告 | 系统 | 机器人主动推送 | 每周一/每月 1 日推送周报月报 |
| 深度分析 | 用户 | 对话 → 机器人 → 必要时打开仪表盘 | 「帮我分析一下这个月为什么飞书工资线逾期率高」 |

### 1.3 对话交互设计

机器人不是冷冰冰的命令行，是有上下文记忆的对话体：

```
用户：「帮我看看这周还有什么没做完的」

机器人：
📋 本周剩余待办（5.14-5.17）
├─ 🔴 P0 今天到期
│   └─ 励步暑期招生方案终稿（今天 18:00）
├─ 🟡 P1 本周内
│   ├─ 斯坦星球课程大纲修订（周五）
│   └─ 诺伊曼产品需求评审（周五）
└─ 💡 建议：今天先集中搞定励步招生方案，还剩 6 小时
    周五的两条建议明后天分两天各一条处理
```

### 1.4 管理层级配置

你提供一个管理层名单（飞书用户 ID + 直属上级 + 角色），系统据此构建升级路由表。格式示例：

```yaml
管理层:
  - id: shaowei
    name: 少巍
    role: 总经理
    reports_to: null  # 顶层，不需升级到别人
    tags: [励步, 斯坦星球, 飞书工资, 诺伊曼]

  - id: zhangsan
    name: 张三
    role: 励步业务负责人
    reports_to: shaowei
    tags: [励步]

  - id: lisi
    name: 李四
    role: 斯坦星球业务负责人
    reports_to: shaowei
    tags: [斯坦星球]
```

### 2.2 需要什么东西、每样是干什么的

不是说你要亲自去选这些技术——而是说每样东西对应解决一个问题。你只需要判断**这个问题你要不要解决**。

#### 必须有的（缺了系统跑不起来）

**① 一台云服务器**
- 干什么：让你的系统 7×24 小时在线，跟你 Mac 死活没关系
- 没有行不行：不行。本地跑的话合盖就停，等于没做
- MVP 成本：阿里云/腾讯云最便宜的轻量服务器，50-100 元/月
- 需要你决定：用阿里云还是腾讯云？（哪个有现成账号用哪个）

**② 一个数据库**
- 干什么：存所有 TDL 数据、管理层名单、催收记录
- 没有行不行：不行。没地方存数据
- MVP 方案：PostgreSQL——开源免费，够用到你公司上市
- 你不需要选，这是默认项

**③ 飞书机器人**
- 干什么：你跟系统对话的入口、系统给你发提醒的出口
- 没有行不行：不行。这是你唯一的交互界面
- 需要你决定：需要一个飞书管理员账号去创建机器人应用

**④ 一个大模型 API**
- 干什么：分析你的 TDL、拆解任务、生成摘要、对话理解
- 没有行不行：不行。没 AI 就等于只有个闹钟
- 需要你决定：主力用 Claude 还是 ChatGPT？
  - Claude：分析能力强，中文理解好，你本来就用的生态
  - ChatGPT：Agent 自动化能力更强
  - 或者两个都要（Claude 做主分析，ChatGPT 做流程自动化）

#### MVP 可以先不要的（以后需要了再加）

**⑤ 备用模型（Hermes / OpenClaw）**
- 干什么：主模型挂了、限流了、或者处理敏感内部数据时做备胎
- 没有行不行：MVP 阶段可以没有。Claude/GPT 挂了最多暂时用不了，不会丢数据
- 什么时候加：系统用起来、团队依赖上了、发现 API 确实偶尔不稳定时再加

**⑥ Web 可视化仪表盘**
- 干什么：在网页上看长期趋势图、行为分析图、团队对比图
- 没有行不行：MVP 可以没有。飞书推送周报月报已经能满足 80% 需求
- 什么时候加：你自己用了 2-4 周后觉得"光看文字不够直观"时再加

**⑦ 钉钉接入**
- 干什么：让用钉钉的管理层也能用
- 没有行不行：MVP 完全不需要。飞书跑通再说
- 什么时候加：推广期，有钉钉用户要接入时

#### 决策清单 ✅ 已确认

| # | 问题 | 决定 |
|---|------|------|
| 1 | 云服务器 | 阿里云（已有账号） |
| 2 | 主力 AI 模型 | Claude + ChatGPT 双模型 |
| 3 | 飞书管理员权限 | 少巍自己持有 |

### 2.3 MVP 技术简化

MVP 阶段只保留核心链路：

```
飞书机器人消息 ←→ FastAPI ←→ PostgreSQL
                       ↓
                  Claude API（TDL 分析 + 对话）
                       ↓
                  Redis（催收定时任务）
```

去掉的（MVP 后补）：
- 向量数据库（先用 PostgreSQL JSON 字段存行为数据）
- Web 仪表盘（先用飞书消息推送代替）
- Hermes/OpenClaw 备用模型
- 钉钉接入
- 审批流/文档深度嵌入（先只做日历+消息）

### 2.4 云端部署方案

| 环境 | 配置建议 | 月成本估算 |
|------|---------|-----------|
| MVP | 2C4G 轻量云服务器 | ~50-100 元/月 |
| 推广期 | 4C8G + 数据库独立 | ~300-500 元/月 |
| 成熟期 | 集群化 + 异地容灾 | ~1000-2000 元/月 |

### 2.5 为什么不能省云端部署（再次强调）

```
你的 Mac                   云端服务器
┌──────────┐              ┌──────────────┐
│ 合盖休眠  │              │ 7×24 运行     │
│ 出门没电  │     VS       │ 跟你的设备无关  │
│ 系统更新  │              │ 定时提醒照发   │
│ 关机下班  │              │ 飞书机器人永远在线│
└──────────┘              └──────────────┘
```

Agent 负责"脑子的思考"，云端负责"永远在线的身体"——两者不可互相替代。

# MVP 实施路线图

## 总览

从零到少巍自己用起来的完整路径，分四个阶段。每个阶段有明确的交付物和验收标准。

```
第 0 阶段        第 1 阶段         第 2 阶段        第 3 阶段
环境搭建        核心开发         自我验证         推广准备
 1-2 天         2-3 周          2-4 周          1-2 周
   ↓               ↓               ↓               ↓
 复用现有ECS    TDL引擎+AI      少巍自己用       开放管理层
 飞书机器人     +催收+反馈      跑通全流程       修bug+优化
 数据库就绪     飞书打通         迭代打磨
```

---

## 第 0 阶段：环境搭建（预计 1-2 天）

**目标**：云端基础设施就绪，飞书机器人能收发消息。

### 要做的事

**0.1 复用现有阿里云 ECS（已就绪）**

你已经有服务器在跑励步钉钉 H5 项目：
```
阿里云 ECS (182.92.9.69)
├─ /opt/bots/libu/backend     ← 励步后端（Python + systemd）
├─ /var/www/libu-h5/dist      ← 励步前端（Nginx）
├─ libu-backend.service       ← 现有服务
└─ Nginx + 域名 libu-h5.bdteach.cn
```

TDL 系统部署到同一台服务器，互不影响：
- 项目路径：`/opt/bots/tdl/backend`
- 服务名：`tdl-backend.service`
- 域名：`tdl.bdteach.cn`（或用现有域名的子路径）
- 路由器由 Nginx 区分，不会跟励步项目冲突

**0.2 飞书机器人应用**
- 飞书开发者后台创建企业自建应用
- 开通权限：消息推送、机器人、日历读取、审批读取
- 配置事件订阅（Webhook 回调地址指向阿里云服务器）
- 获取 App ID + App Secret
- 测试：机器人能收到消息、能回复消息

**0.3 数据库**
- PostgreSQL 容器部署
- 创建 TDL 核心表（按 Part 1 数据模型）
- 创建管理层级表

**0.4 模型 API**
- 注册 Claude API Key
- 注册 OpenAI API Key
- 配置模型路由（Claude 做分析，ChatGPT 做流程编排）

### 验收标准
- [ ] 飞书给机器人发「你好」，机器人回复「你好」
- [ ] 数据库能写入一条测试 TDL
- [ ] Claude API 和 OpenAI API 都能正常调用
- [ ] `tdl-backend.service` 正常运行，与 `libu-backend.service` 互不干扰

## 第 1 阶段：核心开发（预计 2-3 周）

**目标**：催收 + 处理 + 反馈三个模块核心功能可运行。

### 1.1 TDL 引擎（3-4 天）

要做的事：
- TDL CRUD API（创建、查询、更新、删除、状态变更）
- 状态机流转（新建 → AI已分析 → 待处理 → 进行中 → 已完成/已逾期）
- 飞书对话创建 TDL（自然语言 → 结构化存储）
- TDL 查询（「我今天有哪些」「励步这周还有什么」「所有逾期任务」）

### 1.2 AI 处理管道（4-5 天）

要做的事：
- Step 1-2：意图识别 + 字段自动填充（业务线、优先级、职能域）
- Step 3：智能拆解子任务
- Step 4：优先级排序 + 冲突检测
- Step 5：补充提醒（缺截止时间追问、同日多 P0 冲突预警）
- Step 6：飞书消息格式化输出

### 1.3 催收模块（3-4 天）

要做的事：
- 定时任务调度器（每日/每周/每月/每季/每年触发）
- 渐进式催收规则引擎
- 升级链路由（按管理层级关系）
- 飞书主动推送（提醒/逾期/升级通知）

### 1.4 反馈模块（3-4 天）

要做的事：
- 周报/月报自动生成
- 飞书消息推送
- 基础行为统计（完成率、逾期率、响应时间）
- 行为档案初始存储

### 1.5 飞书深度集成（2-3 天）

要做的事：
- 日历事件读取（关键词匹配提取 TDL）
- 消息意图识别（@你 + 待办关键词）
- 命令行式快捷操作（「完成斯坦星球那条」→ 标记完成）

### 验收标准
- [ ] 少巍在飞书里创建一条 TDL，AI 自动分类、拆解、排序
- [ ] 到期 TDL 收到催收提醒
- [ ] 逾期 1 天通知升级
- [ ] 每周一收到周报推送

## 第 2 阶段：自我验证（预计 2-4 周）

**目标**：少巍自己每天用，把所有真实 TDL 都丢进去，边用边调。

### 要做的事

- 每天真实使用（不测试、真干活）
- 记录问题和不顺手的地方
- 调整分类体系（业务线/职能域/关键动作的枚举值）
- 调整催收节奏（提醒频率、时间点、通知语气）
- 打磨 AI 处理质量（拆解准不准、排序合不合理、建议有没有用）
- 管理层名单录入（准备推广）

### 验收标准
- [ ] 连续 2 周每天使用，系统没挂过
- [ ] 催收/处理/反馈三个模块都跑通了
- [ ] 你觉得"这玩意确实能帮我省心"
- [ ] 揪出并修掉至少 10 个不顺手的地方

---

## 第 3 阶段：推广准备（预计 1-2 周）

**目标**：平滑开放给 10-30 人管理层，不出乱子。

### 要做的事

- 录入管理层名单 + 层级关系
- 权限隔离（每个人只看到自己的 TDL，上级可以看到下级的）
- 写使用指南（飞书机器人怎么用，3 分钟能看完的那种）
- 分批开放（先 3-5 人试 1 周 → 再全量）
- 补充备用模型 Hermes/OpenClaw
- 补充 Web 仪表盘（如需要）
- 考虑钉钉接入（如需要）

### 验收标准
- [ ] 首批 3-5 人用起来没大问题
- [ ] 没人说「这玩意不好用，太麻烦」
- [ ] 使用指南能被看懂

## 工作量与分工

| 角色 | 负责 |
|------|------|
| 少巍 | 决策 + 真实使用 + 反馈 + 管理层名单 |
| 开发者（AI Agent 可完成大部分） | 代码开发 + 部署 + 联调 |
| 飞书管理员（少巍） | 飞书应用配置 + 权限审批 |

第 0 + 第 1 阶段的核心开发工作量，用 Claude/ChatGPT Agent 辅助开发，预计 2-3 周可以完成。

---

## 风险与对策

| 风险 | 概率 | 对策 |
|------|------|------|
| 飞书 API 限流 | 低 | 消息推送走批量接口，不做高频轮询 |
| Claude/GPT API 不稳定 | 中 | 双模型互为备份，挂了自动切 |
| 催收太频繁被嫌烦 | 高 | 第二阶段你自己用的时候调，找到舒服的频率 |
| 管理层不习惯用机器人 | 中 | 你带头用，他们看到效果自然会跟 |
| TDL 录入阻力大 | 中 | 飞书自动抓取减少手动录入量，手动录入只需一句话 |

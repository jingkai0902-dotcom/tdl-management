# 06 · MVP 实施方案

> 与 03 的关系：03 定义了五模块全貌，06 定义 MVP 只做前四个、第五个数据层预留。
> 与 07 的关系：07 把这里的范围拆成可执行的开发批次。

---

## 一、MVP 目标

**做一个嵌在飞书里的可靠管理层待办秘书。** 少巍自己用它 2-4 周，证明三件事：

1. 录入够轻——不需要专门花时间录系统
2. 提醒可信——该提醒的时候不遗漏
3. 升级不伤人——催了但不让人反感

---

## 二、MVP 做什么（四个模块）

```
                    MVP 范围
记下来 ──→ 理清楚 ──→ 推得动 ──→ 看得见 ──→ 长出来（数据层预留）
  ✅          ✅          ✅         ✅              🔮
```

### 模块一：意图捕获

- 飞书机器人自然语言创建 TDL
- TDL 创建后自动写入负责人日历（日程事件，不需要员工手动建）
- 低风险自动创建 + 高风险草稿确认

### 模块二：结构化处理

- AI 抽取字段 + 打标签（对照标签词典）
- 缺字段追问（只在必要时问）
- 置信度 < 70% 标「待确认」
- 简洁下一步建议

### 模块三：进度推进

- 每日 08:30 到期/逾期提醒
- 逾期 2 天追问（带按钮：已完成/延期/需协助）
- 逾期 3 天未回复 → 通知上级 + 审计日志
- 逾期 7 天 → 升级
- 暂缓功能（"明天再提醒"）

### 模块四：复盘可见

- 每周一 09:00 飞书周报卡片
- 只做事实账本：新增/完成/逾期/延迟/等待/阻塞/下周到期
- 超过 3 天未推进的 TDL 清单

---

## 三、MVP 不做什么

以下明确排除在 MVP 之外，已确认推迟：

| 不做的事 | 原因 |
|---------|------|
| Web 可视化仪表盘 | 飞书卡片周报够用了 |
| 长期行为记忆与画像 | 数据量不够时是噪音 |
| 语音交互 | MVP 先跑通文本 |
| 钉钉接入 | 飞书跑通再说 |
| OKR 自动对齐检查 | 没有真实维护的 OKR 数据源 |
| 审批流/文档深度嵌入 | 平台工程，非 MVP 核心 |
| 团队公开排名 | 推广后才需要考虑 |
| 深度行为分析 | 精力曲线、拖延模式——数据量不够 |
| 复杂日历联动 | 仅做单向写入（TDL→日历），不做双向同步 |
| 管理层全员推广 | MVP 只有少巍自己用 |
| 会议纪要转 TDL | 🔮 第 2 阶段：用户贴入纪要文本，AI 拆决议→任务 |
| 老板督导航面板 | 🔮 第 2 阶段：本周必盯 N 件事 + 管理者风险名单 |
| 管理者能力成长建档 | 🔮 第 2 阶段：沉淀进化模块的数据维度之一 |
| TDL 模板结构校准 AI | ✅ 已记录：模板三层目标（业务→管理→成长）作为 AI 分类参考 |

---

## 四、模块五的 MVP 处理

沉淀进化模块在 MVP 不开发功能，但从第一天起数据库就为它预留结构：实际耗时、延期次数、关联的业务成果字段。数据从 MVP 第一天就开始积累，到第 2 阶段上线时直接可用。

---

## 五、权限（MVP 简化版）

| 角色 | 权限 |
|------|------|
| 少巍（唯一用户 + 管理员） | 全局查看、创建、修改、配置 |

推广后启用多角色权限（任务负责人/创建人/协作人/直属上级/管理员），数据库表结构已预留。

---

## 六、催收升级（MVP 特殊处理）

MVP 只有少巍一个人用，没有「上级」这个概念。逾期 3 天的升级通知发给少巍自己，推送注明 `[升级]` 标记。推广后按 `reports_to` 字段自动路由。

推广后升级策略按任务类型可配：

| 任务类型 | 升级策略 |
|---------|---------|
| P0 关键任务 | 逾期 1 天可自动升级 |
| P1 管理任务 | 先追问，未回复再升级 |
| P2/P3 普通任务 | 默认不升级，进周报 |
| @delegated 委派 | 先提醒被委派人，抄送创建人 |
| @blocked 阻塞 | 推送语气"需要支持"，非"逾期问责" |

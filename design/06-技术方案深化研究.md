# TDL 系统 — 技术方案深化研究报告

> 基于社区最新实践（截至 2026 年 5 月）的五维度技术调研

## 一、MCP 接口：用在哪、怎么选

### 现状

2025 年是 MCP 在钉钉/飞书爆发的一年，npm 上有成熟的 MCP Server：
- `@china-mcp/feishu-mcp`：8 个工具（消息、文档、日历、任务）
- `@china-mcp/dingtalk-mcp`：双模式（群机器人 + 企业内部应用）

### 2026 年 3 月转折：CLI 取代 MCP-first

钉钉发布「悟空」平台全面 CLI 化，飞书开源 `lark-cli`（2500+ API），行业共识从 MCP-first 转向 CLI-first。核心数据对比：

| | CLI | MCP |
|--|-----|-----|
| Token 消耗 | 1,365 | 44,026（32 倍） |
| 月成本（1 万次操作） | ~$3.2 | ~$55.2（17 倍） |
| 调用成功率 | 100% | 72% |
| 调试 | `--help` 一键 | JSON 传输日志解析 |

### TDL 系统的建议

**不是二选一，是各用所长：**

| 场景 | 方案 | 理由 |
|------|------|------|
| 飞书内部操作（发消息、读日历、创建文档） | **飞书 CLI** | Token 省 32 倍，成功率 100% |
| 钉钉内部操作（用户查询、任务联动） | **钉钉 CLI** | 同上 |
| 跨平台标准化（未来飞书↔钉钉互通） | **MCP** | MCP 的跨厂商标准化能力是 CLI 不具备的 |
| AI Agent 分析处理链路 | **MCP** | 处理管道各步骤作为 MCP tool 暴露，可被任意 Agent 调用 |

**具体落地策略**：
- MVP 阶段：飞书 CLI 做集成（因为你先跑飞书单平台）
- 推广期加钉钉：钉钉 CLI 接入
- 跨平台数据互通：通过 MCP 做标准化抽象层

## 二、钉钉用户 ID 实现团队 TDL 联动

### 钉钉 userId 体系

- 钉钉连接器已从 `unionId` 迁移到 `userId`
- 获取路径：管理后台 → 通讯录 → 成员管理 → 查看 UserID
- 配套字段：`corpId`（组织）、`deptId`（部门）、`agentId`（应用）

### 团队联动方案

```
管理层名单配置
├─ userId: shaowei → 上级: null
├─ userId: zhangsan → 上级: shaowei, 部门: dept_励步
├─ userId: lisi → 上级: shaowei, 部门: dept_斯坦星球
```

基于此实现：
- **通讯录同步**：钉钉通讯录 API 一键导入管理层 → 自动生成升级路由表
- **任务指派**：TDL 创建时 `@某人` → 自动匹配 userId → 写入负责人字段
- **催收升级**：逾期触发 → 查 userId 的 reports_to → 自动通知上级
- **消息推送**：工作通知 API 按 userId 定向推送（不走群聊，私密送达）

### 权限点要求

钉钉对第三方应用有严格权限管理（2025 新规）：
- 通讯录管理权限：必接
- 消息通知权限：必接
- 待办任务权限：有审批场景必接
- 权限使用率保持 > 80%，否则回收

**TDL 系统需要申请的钉钉权限**：通讯录读取 + 工作通知 + 待办任务创建/更新

## 三、社区成熟方案与可借鉴思路

### 最值得关注的 5 个项目

| 项目 | 核心思路 | 对 TDL 系统的参考价值 |
|------|---------|---------------------|
| **OpenAI Symphony** | Kanban 看板 → Agent 自动调度的控制平面 | 正式验证了「任务板驱动 Agent」范式，TDL 本质就是管理者的 Kanban |
| **Corellis GoalOps** | 给一句目标 → 自动拆 6 个子目标、36 张任务卡 | **催收→处理 的自动化拆解**可借鉴其目标分解引擎 |
| **Taskr** | Claude Code 原生 + GitHub Projects 集成 + devlog 持久记忆 | **最接近 TDL 系统的开源实现**，MIT 协议可直接参考架构 |
| **Mergecrew** | 人类审批是不可配置的铁律 | TDL 的**升级催收链**应设计为不可绕过的硬约束 |
| **Khaos Brain** | 预测性记忆，sleep/dream 周期维护 | TDL 的**反馈模块长期记忆**可借鉴其记忆巩固机制 |

### 五个可复用的架构模式

**模式 1：目标分解而非任务分派**
Corellis 的做法是给控制器一句话目标 → 自动拆解 → 分配 → 跟踪。TDL 系统里，管理者说「Q3 斯坦星球招生目标达成」，系统应自动拆出子 TDL 链（市场获客 → Demo 转化 → 关单 → 开班），而非等管理者一条条录。

**模式 2：多层 Agent 层级**
APM 项目用 Planner → Manager → Worker 三层结构。TDL 可对标：
- Planner = 你的战略 TDL（季度/年度）
- Manager = 模块负责人（催收引擎、处理引擎）
- Worker = 具体执行 Agent（一条 TDL 的处理）

**模式 3：持久化记忆 + 自我改进**
Khaos Brain 的 sleep/dream 周期：每天结束时 Agent 主动整理今天的经验 → 合并相似 → 标记失效 → 更新记忆库。TDL 的反馈模块最需要这个——不是被动等你看报告，而是系统主动消化、进化。

**模式 4：成本硬上限**
Mesa 项目强制每个 Agent 每日 token/成本预算，超了直接暂停。TDL 系统跑 AI 处理管道时必须有这个，避免"某天 AI 嗨了烧掉 $50"。

**模式 5：人类审批不可绕过**
Mergecrew 的 production promotion 永远是人工决策。TDL 的催收升级（通知上级、公开逾期数据）必须有人工确认点，不能 AI 自己决定升级。

## 四、哪些可以包装成 Agent 自动化 + Skills 标准化

### Skills 标准化架构（三层）

```
TDL Skill 包结构
├── SKILL.md          # YAML frontmatter(触发条件) + Markdown 指令(核心流程)
├── scripts/          # 执行脚本
├── references/       # 按需加载的参考文档（标签词典等）
└── templates/        # 输入输出模板
```

### 可包装的 6 个 Skill

| Skill | 对应模块 | 设计模式 | 触发条件 |
|-------|---------|---------|---------|
| **tdl-intake** | 处理-Step1-2 | 决策树 | 检测到新 TDL 输入 |
| **tdl-decompose** | 处理-Step3 | 线性流程 | tdl-intake 完成后 |
| **tdl-prioritize** | 处理-Step4 | 决策树 + 循环迭代 | tdl-decompose 完成后 |
| **tdl-remind** | 催收 | 多阶段+检查点 | Cron 定时触发 |
| **tdl-escalate** | 催收升级 | 决策树 | 检测到逾期 |
| **tdl-reflect** | 反馈 | 循环迭代 | 每周一/每月 1 日触发 |

### 设计质量门禁（每个 Skill 必须满足）

- Frontmatter 完整性：name + description（含触发短语）
- Token 预算：正文 < 5K tokens
- 退出条件：每个循环有可量化检查清单
- 反模式扫描：无单一文件膨胀、无模糊指令
- 负例测试：用故意模糊的输入测试行为

### Agent 自动化的关键原则

1. **不要假设 LLM 有"常识"**——每步都写清楚，包括"显而易见"的
2. **借口反驳表**——前置声明 LLM 可能偷懒的路径并反驳
3. **量化阈值**——不说"足够好"，说"完成率 > 80%"
4. **Sandbox 隔离**——敏感操作（升级通知上级）需人类确认

## 五、语音交互：GPT-Realtime-2 的应用

### 模型能力（2026.5.8 发布）

- **GPT-5 级推理能力**原生融入语音回路
- **128K token 上下文**（前代 32K）
- **Preambles**：调用工具时自然说"让我查一下"，不沉默等待
- **并行工具调用**：同时调多个后端工具并告知进度
- **打断处理**：支持中途纠正
- **推理强度可调**：五档（minimal~xhigh），低延迟场景用 low

### TDL 系统的语音交互场景

| 场景 | 当前（文本） | 语音后 |
|------|------------|--------|
| 创建 TDL | 打开飞书 → 打字 | 对手机/电脑说：「下周三前审核斯坦星球暑期班方案，教研组配合」 |
| 查待办 | 打字"今天有什么" | 开车时问：「励步这周还有哪些没做完的」 |
| 更新状态 | 打字"标记完成" | 做完直接说：「斯坦星球课程方案审核搞定了」 |
| 催收提醒 | 文字推送 | 语音播报：「少巍，励步招生方案今天截止，还剩 6 小时」 |

### 降低录入压力的关键点

你提到的「纯打字会给人造成录入压力」——这是真问题。管理层最不乐意做的事就是"往系统里录东西"。语音直接解决了这个：

- 边说边自动转结构化 TDL（利用 GPT-Realtime-2 的并行工具调用能力，一边听你说一边调后端填字段）
- 飞书语音消息 → 机器人接收 → 转文字 → 走处理管道
- 催收提醒也可以语音播报（飞书机器人语音消息）

### 集成路径

MVP 先不上语音（增加复杂度），但架构上预留接口：

```
输入适配层
├─ 飞书文本消息 → 处理管道（MVP）
├─ 飞书语音消息 → 转文字 → 处理管道（推广期）
└─ GPT-Realtime-2 直连 → 实时语音对话（成熟期）
```

## 六、综合建议：MVP 调整项

基于以上调研，对原 MVP 路线图的调整建议：

| # | 调整项 | 原方案 | 调整后 |
|---|--------|--------|--------|
| 1 | 飞书集成方式 | 飞书 SDK | **飞书 CLI**（成本低 17 倍，可靠性高） |
| 2 | TDL 处理管道架构 | 单体 | 拆成 **6 个独立 Skill**，可独立测试和迭代 |
| 3 | 催收升级 | AI 自动决定 | 加 **人类确认闸门**（Mergecrew 模式） |
| 4 | 成本控制 | 无 | 加 **每日 token 预算硬上限**（Mesa 模式） |
| 5 | 长期记忆 | PostgreSQL JSON | 参考 **Khaos Brain** 做周期性记忆巩固 |
| 6 | 语音 | 不考虑 | 架构预留接口，**推广期接入** |
| 7 | 任务拆解 | 手工编码规则 | 参考 **Corellis GoalOps** 做 LLM 驱动的目标分解 |

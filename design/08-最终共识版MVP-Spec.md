# 最终共识版 MVP Spec

> 经少巍、Claude、Codex 三方多轮碰撞后的收敛版本。
> 原则：录入够轻、提醒可信、升级不伤人。

---

## MVP 范围：5 个交付物

```
TDL Intake  →  TDL Core  →  TDL Assistant  →  TDL Reminder  →  TDL Review
 (录入)         (数据)        (AI处理)          (催收)           (反馈)
```

---

## 交付物 1：TDL Intake（录入）

### 1.1 飞书文本录入
- 用户在飞书给机器人发自然语言
- AI 抽字段 → 判断是自动创建还是草稿确认

### 1.2 轻量日历抓取（保留在 MVP，但范围严格收敛）
| 限制项 | 范围 |
|--------|------|
| 读取范围 | **仅少巍本人的日历**（其他人推广后再说） |
| 时间窗口 | 仅未来 7 天 |
| 识别规则 | 标题/描述含 `TDL/待办/需完成/截止/交付` 的事件 |
| 入库策略 | 日历来源**全部草稿，不自动创建** |
| 防重复 | 记录 `source_event_id` |
| 周期处理 | 暂不处理 recurrence，每次只读单实例 |
| 变更处理 | 不做日历→TDL 联动更新（日程改了 TDL 不改） |
| 取消处理 | 日历取消 → TDL 状态标为「已取消」 |
| 隐私保护 | 仅识别含关键字的，安静忽略私人事项 |

### 1.3 确认策略（低风险自动 / 高风险草稿）
```
自动创建条件（全部满足）：
  ✅ 截止时间明确
  ✅ 负责人明确（默认少巍自己）
  ✅ 标题明确
  ✅ AI 置信度 ≥ 85%
  ✅ 不涉及通知他人
  ✅ 不涉及委派他人
  → 发「已创建，可修改」卡片

草稿确认条件（任一满足）：
  ⚠️ 缺截止时间
  ⚠️ 缺负责人
  ⚠️ 置信度 < 85%
  ⚠️ 来自日历自动抓取
  ⚠️ 涉及委派/通知他人
  → 发草稿卡片，用户确认后入库
```

## 交付物 2：TDL Core（数据与权限）

### 2.1 数据库字段（全部建表，三层展示）

**必须字段**（卡片展示）：
`title`, `owner_id`, `due_at`, `status`, `source`, `created_by`, `priority`

**强烈建议字段**（卡片展示）：
`completion_criteria`, `waiting_for`, `blocked_by`

**后台字段**（卡片不展示，AI/系统使用）：
`tdl_id`, `parent_id`, `start_at`, `snooze_until`, `confidence`, `audit_log`, `source_event_id`

### 2.2 状态机
```
draft → active → done
  ↓        ↓
cancelled  overdue → snoozed → active
               ↓
            escalated
```
`draft`：待确认的草稿
`active`：进行中
`overdue`：已逾期（系统自动标）
`snoozed`：用户暂缓（恢复时回到 active）
`escalated`：已升级通知上级
`done`：完成
`cancelled`：取消

### 2.3 权限模型（MVP 只有少巍一人，但建表时预留）

| 角色 | 权限 |
|------|------|
| 任务负责人 | 查看、更新、完成、延期自己的 TDL |
| 创建人 | 查看自己创建的 TDL，催办，关闭 |
| 协作人 | 查看相关 TDL，补充进展 |
| 直属上级 | 只看被升级/逾期/阻塞的下级 TDL |
| 管理员（少巍） | 全局查看与配置 |

### 2.4 审计日志（每次关键动作写入）
创建、AI修改字段、用户确认、延期、完成、催收、升级、上级查看、人工覆盖AI判断。

---

## 交付物 3：TDL Assistant（AI 处理）

MVP 只做四件事，不做 OKR 对齐检查。

### 3.1 字段抽取
从自然语言提取结构化字段，填充标签词典中的业务线、职能域、优先级。

### 3.2 缺失信息判断
检测缺截止时间、缺负责人、缺完成标准时追问。`completion_criteria` 只在 AI 判断"完成标准不清楚"时才问。

### 3.3 标签分类
自动打标签，置信度 < 70% 标"待确认"不硬塞。

新增横向标签：
- `@waiting_for`：等待某人反馈
- `@blocked`：有阻塞
- `@next_action`：下一步明确动作
- `@review`：需要审核
- `@delegated`：已委派
- `@recurring`：周期任务

### 3.4 下一步动作
给出简洁的一句话建议（不是长篇分析）。

子任务拆解限制：只有用户明确说"方案/项目/规划/落地/推进"时才建议拆解。不每条都拆。

## 交付物 4：TDL Reminder（催收）

### 4.1 提醒节奏
```
每日 08:30 → 到期提醒 + 逾期提醒
每周一 09:00 → 本周到期汇总
```

### 4.2 升级策略（MVP 少巍自用期强规则，推广后可配置）

**少巍自用期默认节奏：**
```
逾期 1 天 → 提醒本人
逾期 2 天 → 追问本人（飞书卡片按钮：已完成 / 延期 / 需协助）
逾期 3 天 → 本人未回复 → 通知上级 + 审计记录
逾期 7 天 → 升级到少巍
```

**推广后按任务类型可配：**
| 任务类型 | 升级策略 |
|---------|---------|
| P0 当日关键任务 | 可自动升级 |
| P1 管理任务 | 先追问，未回复再升级 |
| P2/P3 普通任务 | 默认不升级，进入周报 |
| @delegated 委派 | 先提醒被委派人，抄送创建人 |
| @blocked 阻塞 | 推送语气"需要支持"，非"逾期问责" |

### 4.3 升级语气
不回消息说「逾期」，说「需要支持」。系统推送上说「X 任务可能需要关注」而非「X 任务逾期未完成」。

### 4.4 暂缓
用户说"明天再提醒"→ `snooze_until` = 明天 09:00，到时恢复。

## 交付物 5：TDL Review（反馈）

### 5.1 周报（每周一 09:00 飞书卡片推送）
只做**事实账本**，不做行为画像：

```
📊 TDL 周报（5.11-5.17）
├─ 本周新增：15 条
├─ 本周完成：12 条（80%）
├─ 逾期未清：3 条
├─ 延期处理：2 条
├─ 等待他人：4 条（@张三 2条，@李四 1条…）
├─ 阻塞中：1 条
├─ 超过 3 天未推进：2 条
│   ├─ 励步招生方案终稿（5 天未动）
│   └─ 斯坦星球课程大纲修订（4 天未动）
├─ 下周到期：6 条
└─ 卡点提示：飞书工资相关 TDL 连续 2 周有阻塞
```

不做：高效时段分析、拖延模式识别、精力曲线、行为画像。

## 工程约定

### 配置集中管理
硬编码参数放配置文件，不放 Skill 文档里：
```
config/
├── tdl_rules.yaml         # 置信度阈值、自动创建条件
├── tag_dictionary.yaml    # 标签词典（从 design/05 生成）
├── escalation_policy.yaml # 升级策略参数
└── feishu_config.yaml     # 飞书应用配置
```

### Skills 指令层
6 个 `SKILL.md` 写行为原则，不写硬编码参数。AI prompt、后端函数、测试用例都读同一份 config。

### AI 护栏
- **高风险护栏**：升级通知、批量提醒、修改他人任务 → 必须人工确认
- **输出护栏**：AI 不允许编造不存在的业务线/人员/日期
- **成本护栏**：每人每日 token 上限

### 飞书集成
- 生产后端：飞书官方 SDK（稳定、错误处理完备）
- 调试运维：飞书 CLI（快速验证）

## 不做清单（明确排除）

以下不在 MVP，讨论阶段已确认推迟：

- Web 可视化仪表盘
- 长期行为记忆与画像
- 语音交互（架构预留接口）
- 钉钉接入
- OKR 自动对齐检查
- 审批流/文档深度嵌入
- 团队公开排名
- 深度行为分析（精力曲线、拖延模式）
- 复杂日历周期处理
- 管理层全员推广

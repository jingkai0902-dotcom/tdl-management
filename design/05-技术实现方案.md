## 一、整体结构

系统跑在阿里云 ECS 上，通过钉钉机器人与管理层交互。你的 Mac 不跑任何服务——合盖不影响。

```
钉钉平台（主交互）
  ├─ 机器人消息 ──→ 对话、文本 TDL、会议纪要输入
  ├─ 卡片推送 ←─── 提醒、周报、确认卡片
  └─ 日历 ────────→ TDL 自动写入日程（钉钉日历）
        │
        ▼
阿里云 ECS (182.92.9.69)
  ├─ tdl-backend (FastAPI) ──→ OpenAI（主）+ DeepSeek（备）
  ├─ tdl-stream-bot ─────────→ 钉钉 Stream 长连接
  ├─ PostgreSQL ──────────────→ meeting / decision / tdl / audit
  ├─ APScheduler ─────────────→ 定时提醒（按班制路由）
  └─ Nginx ──────────────────→ 反向代理

飞书平台（数据分析）
  └─ 多维表格 ──→ TDL 数据仪表盘（和薪资系统同平台）
```

✅ 已确认：与励步钉钉 H5 项目部署在同一台 ECS，目录分离、服务分离、互不影响。

## 二、用什么技术，分别解决什么问题

### 必须有的

**① 云服务器（已有）**
- 阿里云 ECS 182.92.9.69，已在运行励步项目
- TDL 系统新增目录 `/opt/bots/tdl/backend`
- systemd 服务拆成两个：`tdl-backend.service`（FastAPI + 调度器）和 `tdl-stream-bot.service`（钉钉 Stream 长连接）

**② 后端框架：FastAPI**
- Python 异步框架，适合 AI API 调用密集的场景

**③ 数据库：PostgreSQL**
- 存 meeting / decision / tdl / audit 数据、管理层名单、标签词典

**④ 定时调度：APScheduler**
- 每日提醒、每周周报的定时任务

**⑤ 钉钉集成：Stream 模式（主交互）**
- 钉钉机器人 Stream 模式——WebSocket 长连接，收发消息和卡片
- 你有现成的钉钉开发经验，不需要从零学

**⑥ 飞书集成：多维表格（辅助）**
- 飞书多维表格：TDL 数据仪表盘，和薪资系统同平台
- 复用 starplanet_ai 项目里的 `feishu_client.py` 中可复用的鉴权与请求封装

**⑦ AI 模型：OpenAI + DeepSeek 双模型**

| 角色 | 模型 | 用途 | fallback |
|------|------|------|---------|
| 主模型 | OpenAI（具体型号由配置决定） | 会议纪要拆解、TDL 分类、周报生成 | 失败提示"稍后重试" |
| 备选模型 | DeepSeek（具体型号由配置决定） | 简单摘要、提醒文案、中文场景补充 | 仅非关键任务在主模型不可用时自动切换 |

关键链路（会议纪要拆解、标签分类）不自动降级。API Key 存 `.env`。

**⑧ 日历写入（钉钉日历）**
- 使用钉钉**个人权限**模式，不再走旧版应用级日历接口
- 每位管理者首次授权一次后，系统以其个人身份写入其钉钉日历
- `Calendar.Event.Write` 负责日程写入，`Contact.User.Read` 用于读取当前授权人的 `unionId`
- 授权回调会把 `unionId` 通过 OAPI 解析回通讯录 `userId`，确认“授权人”和 TDL 负责人一致后才落库；日历 OpenAPI 路径使用 `userId`

### MVP 不需要的

| 东西 | 状态 |
|------|------|
| 钉钉 H5 实时语音（Realtime-2） | V1.1 |
| Web 可视化仪表盘 | 用飞书多维表格代替 |
| 备用模型（Hermes/OpenClaw） | 双模型已覆盖 |
| 向量数据库 | 第 2 阶段 |
| OKR 对齐 | 需要真实 OKR 数据源 |

## 四、密钥管理

`.env` 存密钥（不提交 Git），`config/*.yaml` 存业务参数。

```
项目根目录/
├── .env                    ← 不进 Git：OpenAI Key, DeepSeek Key, DB密码, 钉钉 AppSecret, 飞书 AppSecret
├── config/
│   ├── tdl_rules.yaml
│   ├── tag_dictionary.yaml
│   ├── escalation_policy.yaml
│   ├── dingtalk_config.yaml
│   └── feishu_config.yaml
```

日历授权还需要：

- `PUBLIC_BASE_URL`
- `DINGTALK_OAUTH_SCOPE`（默认 `openid Contact.User.Read Calendar.Event.Write`）
- `DINGTALK_OAUTH_REDIRECT_URI`

机器人消息走 Stream 模式，本身不依赖公网 webhook；但个人日历授权回调需要一个公网 HTTPS 地址。

## 五、AI 安全护栏

| 护栏 | 做法 |
|------|------|
| 不编造信息 | 标签词典做约束 |
| 高风险需确认 | 升级通知 → 人工确认后执行 |
| 成本控制 | 每人每日 token 上限 |
| 审计日志 | 每次创建、修改、催收、升级留痕 |

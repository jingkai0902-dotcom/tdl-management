# 针对 Codex 建议的逐条评估与方案调整

## 总体评价

Codex 的核心判断「MVP 偏大，要从 AI 管理平台收缩成可靠待办秘书」——**完全同意**。具体建议大部分采纳，几条有不同意见。

---

## 一、数据模型：Codex 建议的 9 个补强字段

| 字段 | Codex 建议 | 我的判断 | 结论 |
|------|-----------|---------|------|
| `tdl_id` | 稳定唯一 ID | **采纳**。对话引用"那条任务"必须靠这个 | MVP 加入 |
| `start_at` | 开始时间 | **采纳**。只有 deadline 没有起点，优先级排序缺关键信息 | MVP 加入 |
| `blocked_by` | 依赖阻塞 | **采纳**。管理层待办最大痛点：卡在别人手里的事没人跟踪 | MVP 加入 |
| `waiting_for` | 等待谁反馈 | **采纳**。与 blocked_by 不同——不一定阻塞，但需要等 | MVP 加入 |
| `completion_criteria` | 完成标准 | **采纳**。没有这个，周报只能统计"打勾"不能统计"真完成" | MVP 加入 |
| `confidence` | AI 分类置信度 | **采纳**。低置信度进入"待确认"而非硬塞 | MVP 加入 |
| `audit_log` | 操作留痕 | **采纳**。催收升级有争议时需要追溯 | MVP 加入 |
| `parent_id` | 父任务/项目 | **采纳**。大任务拆子任务必须 | MVP 加入 |
| `snooze_until` | 暂缓提醒 | **采纳**。用户说"明天再提醒我"的场景 | MVP 加入 |

**结论：9 个字段全部采纳。** 没有一条是多余的。

## 二、交互模式：确认式卡片 vs 一次性自信创建

Codex 建议「AI 生成草稿卡片 → 用户确认 → 入库」，不要一次性自信创建。

**同意。** 具体设计：

```
用户：「下周三前审核斯坦星球暑期班方案」

机器人返回飞书卡片：
┌────────────────────────────────────────┐
│ 📌 TDL 草稿                             │
│                                        │
│ 斯坦星球暑期班课程方案审核                │
│ ├─ 负责人：@少巍                        │
│ ├─ 截止：2026-05-20 18:00              │
│ ├─ 业务线：斯坦星球 / 教务教学            │
│ ├─ 优先级：P1-重要不紧急                 │
│ ├─ 完成标准：[待补充]                    │
│ └─ 置信度：85%                          │
│                                        │
│ [✅ 确认创建] [📅 改时间] [👤 改负责人]   │
│ [📝 补充标准] [⚡ 改优先级] [🗑 忽略]    │
└────────────────────────────────────────┘
```

这是 MVP 该有的交互体验，不是后期优化项。

## 三、催收升级：需要保护机制，但要防止"软约束"失效

Codex 建议升级前先问负责人：「是否延期/已完成/需协助/确认升级？」

**同意加保护，但有补充。**

问题：如果升级前先问他、他不回复，系统就永远不升级？那催收就失效了。

方案：**追问 + 超时自动升级**。

```
逾期 1 天 → 提醒本人
逾期 2 天 → 追问：「已完成/延期/需协助？」+ 附带按钮
逾期 3 天 → 本人未回复 → 自动通知上级 + 审计日志记录
逾期 7 天 → 升级到少巍
```

追问给了逃生窗口（2天），但不回复不等于逃避。这样既有人性化的缓冲，又有不可绕过的底线。

Codex 引用的 Atlassian Clean Escalations「理性、协作式升级」——核心不是不升级，是升级不是为了惩罚而是为了解决阻塞。这个理念完全对，在设计语言里体现就行（推送语气：「X 任务可能需要支持」而不是「X 逾期，请处理」）。

## 四、AI 架构：先函数链，后 Skill，这点有分歧

Codex 说 MVP 不要上 Agent/Skill 框架，先做可测试的函数链。

**部分同意，部分不同意。**

同意的地方：第一版代码应该做干净的 Python 函数链，可独立测试，不要引入 Agent 编排框架（LangChain/AutoGPT 之类）的重量。

不同意的地方：Skills 不是"框架"。Superpowers 的 Skill 本质上就是一个 Markdown 文件 + 一些规则——它是给 AI 看的**指令文档**，不是给代码看的**框架**。把 6 个 Skill 的 SKILL.md 写好，不等于引入框架复杂度，等于把 AI 的行为规则文档化了。

**折中方案：**

```
MVP 代码层：
  intake_extract() → classify_tags() → ask_missing() → format_card()
  纯 Python 函数链，单元测试覆盖

MVP 指令层：
  6 个 SKILL.md 写清楚每个函数的 AI 行为规则
  不是给代码框架用的，是给调用方 Claude/GPT 看的
```

这样既保持了代码的简洁可测，又保留了 Skill 标准化带来的行为可控性。Codex 说的「框架复杂度早于业务验证」是对的——但前提是你在引入框架。Skills 不是框架。

## 五、CLI / SDK / MCP：Codex 的判断比我原来的结论更准确

Codex 说：生产后端优先用官方 SDK，Agent 辅助运维用 CLI，跨平台抽象后期才用 MCP。

**这次 Codex 对。** 我原报告里说「飞书内部操作用 CLI」，但 CLI 更适合运维/调试，生产后端确实应该走 SDK——错误处理、重试、限流这些 SDK 原生支持，CLI 不适合高频调用。

修正后的分层：

| 层 | 用什么 | 理由 |
|----|--------|------|
| TDL 后端 → 飞书（发消息、读日历） | 飞书官方 SDK | 稳定、错误处理完备、支持高并发 |
| Agent 调试/运维（手动操作飞书） | 飞书 CLI | 轻量、人类可读、快速验证 |
| 未来飞书↔钉钉互通 | MCP | 跨平台标准化 |

## 六、标签词典：横向标签值得加

Codex 建议加 7 个横向标签。逐一判断：

| 标签 | 是否加入 | 理由 |
|------|---------|------|
| `@waiting_for` | ✅ | 管理层最常见状态之一，已在数据模型加了 waiting_for 字段 |
| `@blocked` | ✅ | 已在数据模型加了 blocked_by 字段 |
| `@next_action` | ✅ | GTD 经典概念，催收提醒时有用：「下一步：联系教研组确认时间」 |
| `@review` | ✅ | 需要审核的 TDL，处理策略不同 |
| `@delegated` | ✅ | 已委派但没完成的，催收对象是被委派人而非创建人 |
| `@recurring` | ✅ | 周期任务不需要每次手动创建 |
| `@two_minutes` | ❌ | GTD 的「两分钟原则」是个人效率技巧，管理层待办很少有两分钟能完成的。加上反而让标签体系不严肃 |

GTD 的「收集、澄清、组织、回顾」循环确实适合这个系统——催收模块本质就是 GTD 的「回顾」环节自动化。

## 七、MVP 范围：应该加回日历自动抓取

Codex 建议把日历放到第 2 阶段。**这一点我要反驳。**

理由：

1. **录入阻力是 MVP 成败的关键**。如果管理层每天要打开飞书打字创建 TDL，用不了三天就停了。日历自动抓取让系统**被动接收** TDL——你正常安排日程，系统自动识别。

2. **日历抓取技术成本极低**。飞书日历 API 一个 GET 请求的事，不是什么平台工程。不是「深度嵌入审批流」那个级别的工作量。

3. **没有自动抓取的 MVP = 高级闹钟**。你的核心愿景之一是「自动抓取碎片任务」，如果 MVP 完全没有这一环，整个系统的价值感会弱很多。

建议修正：

| 模块 | Codex 建议 | 我建议 |
|------|-----------|--------|
| 日历自动抓取 | 第 2 阶段 | **第 1 阶段保留**（工作量小、价值大） |
| 审批流/文档深度嵌入 | 第 2 阶段 | 同意推后 |
| Web 仪表盘 | 第 3 阶段 | 同意推后 |
| 长期记忆/行为画像 | 第 3 阶段 | 同意推后 |
| 语音 | 第 3 阶段 | 同意推后 |
| 钉钉 | 推广期 | 同意推后 |

## 八、修正后的 MVP 路线图

### 第 0 阶段不变
复用现有 ECS + 飞书机器人 + 数据库 + 双模型 API

### 第 1 阶段：4 个小闭环（Codex 的框架，我调了内容）

**闭环 1：TDL 录入（文本 + 日历自动抓取）**
- 飞书机器人文本对话创建 TDL → 确认卡片 → 入库
- 飞书日历事件自动抓取（含"待办/TDL/任务"关键词的事件）
- AI 结构化抽取 + 缺字段追问
- 输出的不是文字，是飞书卡片

**闭环 2：AI 处理**
- 标签自动分类（置信度<70% 标"待确认"）
- 子任务拆解建议
- 冲突/遗漏检测
- 管理方法论注入（四象限、OKR 对齐检查）

**闭环 3：催收**
- 每日 08:30 到期提醒 + 逾期提醒
- 逾期 > 2 天追问本人（带按钮：已完成/延期/需协助）
- 逾期 > 3 天本人未回复 → 通知上级（审计日志记录）
- 逾期 > 7 天 → 升级到少巍

**闭环 4：反馈**
- 每周一 09:00 推送周报（飞书卡片）
- 基础统计：完成率、逾期率、按业务线分布

### 第 2 阶段以后
- 日历深度集成（不只是关键词抓取，而是智能关联）
- 审批流/文档嵌入
- Web 仪表盘
- 长期记忆沉淀
- 语音交互
- 钉钉接入
- 开放给管理层团队

---

## 总结

Codex 的判断：「方向值得做，但第一版应该从 AI 管理平台收缩成飞书里的可靠管理层待办秘书」——**这是整轮讨论最有价值的结论。**

采纳了 Codex 的：数据模型 9 个补强字段、确认式卡片交互、升级保护机制、横向标签、函数链优先于框架、SDK 优先于 CLI。

反驳了 Codex 的：日历自动抓取应该留在 MVP（工作量极小价值极大）、Skills 不是框架不会增加复杂度、两分钟标签不适合管理层场景。

修正了原报告一个判断：飞书集成主力用 SDK 不是 CLI。

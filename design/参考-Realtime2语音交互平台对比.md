# Realtime-2 语音交互：钉钉 vs 飞书 平台对比

> 核心问题：GPT-Realtime-2 走 WebSocket 直连 OpenAI，但用户端音频怎么接入？在钉钉还是飞书做？

## 一、技术通道对比

| 维度 | 钉钉 | 飞书 |
|------|------|------|
| 实时通信方式 | Stream 模式（WebSocket 长连接），已成标准 | HTTP 长轮询 + Webhook 回调 |
| 小程序/应用能力 | H5 小程序，支持 WebRTC、麦克风权限 | 飞书小程序，同样支持 WebRTC |
| Realtime-2 接入路径 | 钉钉 H5 → WebRTC 捕获音频 → 后端 WebSocket 转接 Realtime-2 API | 飞书小程序 → WebRTC → 同上 |
| 现有用户侧经验 | 你做过钉钉音频分析应用（销售录音上传 + AI 分析） | 你有飞书 CLI 和管理员权限，但飞书小程序开发经验未知 |
| 语音消息转文字 | 机器人接收语音消息 → 下载 AMR → Whisper，简单成熟 | 类似，飞书语音消息同样支持 |

## 二、用户体验对比

| 维度 | 钉钉 | 飞书 |
|------|------|------|
| 用户打开方式 | 打开钉钉 H5 小程序 → 点击通话 → 说话 | 打开飞书小程序 → 点击通话 → 说话 |
| 管理层接受度 | ✅ 管理层日常使用钉钉，不需要换 App | ⚠️ 管理层不常用飞书（飞书是你和薪资系统用的） |
| 网络环境 | 国内服务器，稳定 | 国内服务器，稳定 |
| 语音通话体验 | 钉钉 H5 可以模拟原生通话界面 | 飞书小程序同样可以 |

## 三、决定性因素

**钉钉胜出，四个理由：**

1. **用户习惯**：管理层主平台是钉钉。让他们在飞书里用语音交互是"多一个平台"，在钉钉里做是"现有平台多一个功能"
2. **你已有钉钉开发经验**：音频分析应用、机器人 Stream 模式你全做过，飞书小程序你没开发过
3. **Stream 模式适配 Realtime-2**：钉钉 Stream 本身就是 WebSocket 长连接，和 Realtime-2 的 WebSocket 协议天然匹配。后端可以做一个 Relay——钉钉 Stream 接收音频 → 转发到 Realtime-2 WebSocket → 返回音频给钉钉
4. **飞书没有语音交互优势**：飞书强在数据/多维表格/CLI，不是语音。你之前也说"飞书的语音目前不太好落地"

**飞书做什么**：飞书做 TDL 数据管理和多维表格仪表盘，和薪资系统同平台——这是飞书的强项。语音交互放钉钉。

## 四、建议

| 角色 | 平台 | 理由 |
|------|------|------|
| 语音交互（Realtime-2 实时对话） | **钉钉** | 用户习惯 + 开发经验 + Stream 模式天然适配 |
| 语音消息兜底 | **钉钉** | 同一平台，自然降级 |
| TDL 数据管理 | 飞书 | 多维表格 + CLI |

不是先路径 A 再路径 B——是直接做 Realtime-2 实时对话，钉钉 H5 小程序承载。语音消息作为降级兜底，同一平台内无缝切换。

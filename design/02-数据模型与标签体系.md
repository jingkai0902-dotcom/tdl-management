# 02 · 数据模型与标签体系

> 与 01 的关系：01 定义了五模块框架，02 定义每个模块操作的数据长什么样。
> 与 03 的关系：03 每讲一个功能时，会引用这里的字段名称。

## 一、一条 TDL 包含什么信息

一条待办事项在系统里的完整数据结构。分三层：基础信息（必填）、分类标签（AI 自动打）、后台字段（系统用，人不看）。

### 1.1 基础信息

| 字段 | 说明 | 示例 |
|------|------|------|
| `title` | 任务简述 | "斯坦星球暑期班课程方案审核" |
| `owner_id` | 负责人飞书 ID | @shaowei |
| `due_at` | 截止时间 | 2026-05-20 18:00 |
| `status` | 当前状态 | draft / active / overdue / done 等 |
| `source` | 来源 | manual / feishu_msg / feishu_calendar |
| `created_by` | 谁创建的 | 飞书 user_id |
| `priority` | 优先级 | P0 / P1 / P2 / P3 |

⚠️ 待你确认：P0-P3 四档优先级够用吗？P0=紧急重要，P1=重要不紧急，P2=紧急不重要，P3=常规——这套 Eisenhower Matrix 的标准分法用在你的管理场景里是否匹配？

### 1.2 分类标签（AI 自动打）

| 字段 | 说明 | 是否必选 | 可选值来源 |
|------|------|---------|-----------|
| `business_line` | 业务线 | ✅ 必选 | 见下方 §2.1 |
| `function_domain` | 职能域 | ✅ 必选 | 见下方 §2.2 |
| `product_line` | 产品线 | 可选 | 见下方 §2.3 |
| `stage` | 流程阶段 | AI 自动判 | 立项 / 执行中 / 验收 / 复盘 |
| `key_actions` | 关键动作 | AI 自动填 | 见下方 §2.4，可多选 |
| `tags` | 横向标签 | AI 自动打 | 见下方 §2.5 |
| `confidence` | AI 置信度 | 系统内部用 | 0.0 ~ 1.0 |

### 1.3 协作与依赖

| 字段 | 说明 |
|------|------|
| `waiting_for` | 等待谁反馈（飞书 user_id） |
| `blocked_by` | 被哪条 TDL 阻塞（引用另一条 TDL 的 ID） |
| `parent_id` | 父任务（大任务拆子任务时用） |

💡 设计考量：`waiting_for` 和 `blocked_by` 是两个不同的概念。一条 TDL 可能不阻塞但需要等（等教研组反馈），也可能被硬阻塞（课程大纲没批之前不能写教案）。分开记录才能做出精准的「卡在哪」分析。

### 1.4 结果与回收

| 字段 | 说明 |
|------|------|
| `completion_criteria` | 完成标准——不是打勾，是做到什么程度算完成 |
| `outcome_kpi` | 效果指标，如"家长续费率提升 5%" |
| `roi_estimate` | 收益预估：高 / 中 / 低 |

💡 设计考量：`completion_criteria` 很重要但不会强迫填——只在 AI 判断"完成标准不清楚"时才追问。平时不出现，不增加录入负担。

### 1.5 系统内部字段（人不看，AI 和后台用）

`tdl_id`（唯一编号，TDL-20260514-001）、`start_at`（开始时间）、`snooze_until`（暂缓到什么时候恢复提醒）、`source_event_id`（日历事件 ID，防重复）、`audit_log`（每次操作记录：谁、什么时候、做了什么）

按职能域拆分，AI 根据 TDL 内容自动匹配。完整列表参考标签词典原文 [查看](computer:///Users/frankj/companyai/TDL管理交互系统/design/05-标签词典.md)，核心动作举例如下：

### 2.5 横向标签

这些标签悬挂在所有分类之上，描述任务与人的关系：

| 标签 | 含义 | 示例 |
|------|------|------|
| `@waiting_for` | 等待某人反馈 | 等教研组回复方案确认 |
| `@blocked` | 有硬阻塞 | 大纲没批不能写教案 |
| `@next_action` | 下一步明确动作 | 联系教研组确认时间 |
| `@review` | 需要审核 | 方案等少巍终审 |
| `@delegated` | 已委派给他人 | 让张三去跟进 |
| `@recurring` | 周期任务 | 每周一周报 |

💡 设计考量：GTD 里强调区分 waiting for 和 next action——前者是被动等待，后者是可以主动推进的下一步。这个区分让催收系统能做出不同的提醒策略（waiting for 是追问对方，next action 是提醒你动手）。

## 三、状态流转

一条 TDL 从诞生到结束的生命周期：

```
draft ──→ active ──→ done
  │          │
  │          ├──→ overdue ──→ snoozed ──→ active
  │          │       │
  │          │       └──→ escalated
  │          │
  │          └──→ cancelled
  │
  └──→ cancelled（草稿 24 小时未确认）
```

| 状态 | 含义 |
|------|------|
| `draft` | AI 生成的草稿，等用户确认 |
| `active` | 进行中 |
| `overdue` | 已逾期，系统自动标记 |
| `snoozed` | 用户暂缓，到期自动恢复 |
| `escalated` | 已升级通知上级 |
| `done` | 已完成 |
| `cancelled` | 取消（草稿未确认或主动取消） |

## 四、数据流向

数据在五个模块之间怎么流动：

```
用户发消息/创建日历事件
        │
        ▼
  [意图捕获]  创建 draft TDL
        │
        ▼
  [结构化处理] AI 打标签、拆解、排序
        │
        ▼
  [进度推进]   按状态和策略发提醒
        │
        ▼
  [复盘可见]   统计完成率、阻塞、等待
        │
        ▼
  [沉淀进化]   长期累积（MVP 后上线）
```

每条 TDL 的状态变更都写入 `audit_log`。谁在什么时候做了什么——可追溯。

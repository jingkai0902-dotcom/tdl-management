## 一、一条 TDL 长什么样

### 1.1 必填字段（创建时必须有的）

| 字段 | 说明 | 示例 |
|------|------|------|
| `title` | 任务简述 | "斯坦星球暑期班课程方案审核" |
| `owner_id` | 负责人（飞书/钉钉 user_id） | @shaowei |
| `due_at` | 截止时间 | 2026-05-20 18:00 |
| `status` | 当前状态 | active / overdue / done 等（见 §三） |
| `priority` | 优先级 | P0 / P1 / P2 / P3 |

### 1.2 AI 自动填充的字段

| 字段 | 说明 | 可选值来源 |
|------|------|-----------|
| `business_line` | 业务线 | 见 §二-2.1 |
| `function_domain` | 职能域 | 见 §二-2.2 |
| `product_line` | 产品线 | 见 §二-2.3，可选 |
| `stage` | 流程阶段 | 立项 / 执行中 / 验收 / 复盘 |
| `key_actions` | 关键动作 | 见 §二-2.4，可多选 |
| `tags` | 横向标签 | 见 §二-2.5，可多选 |
| `confidence` | AI 分类置信度 | 0.0 ~ 1.0，< 0.7 标"待确认" |

### 1.3 协作与依赖

| 字段 | 说明 |
|------|------|
| `waiting_for` | 等谁的反馈（user_id） |
| `blocked_by` | 被哪条 TDL 阻塞（引用另一条 TDL） |
| `parent_id` | 父任务（大任务拆子任务时用） |
| `completion_criteria` | 完成标准——做到什么程度算完成 |

💡 `waiting_for` 和 `blocked_by` 不同：前者是被动等（等教研组回意见），后者是硬阻塞（大纲没批不能写教案）。

### 1.4 结果回收（可选）

| 字段 | 说明 |
|------|------|
| `outcome_kpi` | 效果指标，如"家长续费率提升 5%" |
| `roi_estimate` | 收益预估：高 / 中 / 低 |

### 1.5 系统内部字段

人不看、AI 和后台用：`tdl_id`（唯一编号）、`start_at`、`snooze_until`（暂缓恢复时间）、`source`（manual / dingtalk_msg / feishu_calendar）、`source_event_id`（日历事件 ID，防重复）、`created_by`、`created_at`、`updated_at`、`audit_log`

## 三、一条 TDL 的一生（状态流转）

```
draft ──→ active ──→ done
  │          │
  │          ├──→ overdue ──→ snoozed ──→ active
  │          │       │
  │          │       └──→ escalated
  │          │
  │          └──→ cancelled
  │
  └──→ cancelled（草稿 24 小时未确认自动取消）
```

| 状态 | 含义 |
|------|------|
| `draft` | AI 生成的草稿，等确认 |
| `active` | 进行中 |
| `overdue` | 已逾期，系统自动标记 |
| `snoozed` | 用户暂缓，到时恢复 |
| `escalated` | 已升级通知上级 |
| `done` | 已完成 |
| `cancelled` | 取消 |

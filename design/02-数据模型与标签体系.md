# 02 · 数据模型与标签体系

> 与 01 的关系：01 定义了谁在用、什么痛点，02 定义系统里存什么数据。
> 与 03 的关系：03 的每个功能模块操作的数据结构都来自这里。
> 与 04 的关系：04 定义每个字段在界面上怎么呈现、怎么操作。

💡 文档里的 `title`、`owner_id` 等是**数据库字段名**——给开发者看的。用户实际看到的界面是中文：「任务标题」「负责人」「截止时间」。

---

## 一、数据分层（三层对象）

系统不只是存待办任务——核心用例是"月会产生决议，决议拆成执行动作"。三层分开：

| 对象 | 说明 | 示例 |
|------|------|------|
| `meeting`（会议） | 每次月度复盘会的原始材料、AI 摘要、参会人、版本 | "励步 5 月月度复盘会" |
| `decision`（决议） | 会上形成的决议事项。一条决议可拆出多条 TDL | "绿谷校区续费率目标 80%，时颖负责前端配合" |
| `tdl`（执行动作） | 可追踪的单条待办。`decision_id` 追溯来源 | "时颖：6 月新开班班级数量确认（周五前）" |

💡 三层分开的好处：一次会议的原始材料有地方放；一条决议拆多条 TDL 时能回溯来源；"是否兑现上次承诺"可以稳定查询（对比上次 meeting 的 decision 和本次 meeting 前的完成状态）。

## 二、一条 TDL 长什么样

### 1.1 核心字段（创建时系统必须有，大部分可由 AI 自动填充）

| 字段（DB） | 用户看到 | 说明 | 填写方式 | 示例 |
|-----------|---------|------|---------|------|
| `title` | 任务标题 | 一句话说清楚 | AI 提取，用户可改 | "斯坦星球暑期班课程方案审核" |
| `owner_id` | 负责人 | 谁来完成 | AI 提取；缺失时先保留草稿、待用户确认 | @shaowei |
| `due_at` | 截止时间 | 单日或时间段 | AI 提取，缺了追问 | 2026-05-20；范围：05-18~05-20 |
| `participants` | 参与人 | 协作成员 | AI 提取 | @zhangsan, @lisi |
| `status` | 状态 | 当前阶段 | 系统自动流转 + 用户手动改 | 草稿/进行中/已逾期/已暂缓/已升级/已完成/已取消 |
| `priority` | 优先级 | 重要紧急程度 | AI 自动判，用户可调整 | P0/P1/P2/P3（见 §一末尾） |

💡 填写方式说明：
- **AI 提取**：用户说话时 AI 自动从内容里抓，用户可在卡片上改
- **AI 自动判**：AI 综合判断，用户不用管，除非觉得不准
- **系统自动流转**：系统根据时间和规则自动变，人不操作
- **用户手动改**：用户在卡片上点按钮或打字改

💡 title 不需要模板——你正常说话就行。AI 从自然语言里提取。「下周三前把斯坦星球暑期班方案审完」→ AI 提取为"斯坦星球暑期班课程方案审核"。

### 1.2 AI 自动填充的字段

| 字段 | 说明 | 可选值来源 |
|------|------|-----------|
| `business_line` | 业务线 | 见 §二-2.1 |
| `function_domain` | 职能域 | 见 §二-2.2 |
| `product_line` | 产品线 | 见 §二-2.3，可选 |
| `stage` | 流程阶段 | 立项 / 执行中 / 验收 / 复盘 |
| `key_actions` | 关键动作 | 见 §二-2.4，可多选 |
| `tags` | 横向标签 | 见 §二-2.5，可多选 |
| `confidence` | AI 分类置信度 | 0.0 ~ 1.0，< 0.7 标"待确认" |

### 1.3 协作与依赖

| 字段 | 说明 |
|------|------|
| `waiting_for` | 等谁的反馈（user_id） |
| `blocked_by` | 被哪条 TDL 阻塞（引用另一条 TDL） |
| `parent_id` | 父任务（大任务拆子任务时用） |
| `completion_criteria` | 完成标准——做到什么程度算完成 |
| `linked_goal` | 关联的更大目标/愿景（可选） | "服务于斯坦星球 Q3 产品升级" |

💡 `waiting_for` 和 `blocked_by` 不同：前者是被动等（等教研组回意见），后者是硬阻塞（大纲没批不能写教案）。`linked_goal` 是五层模型第一层"结构性张力"的轻量实现——不是每条 TDL 都要关联目标，但允许标记这条任务在推动什么更大的成果。

### 1.4 结果回收（人工 + AI 共建，可选）

| 字段 | 说明 |
|------|------|
| `outcome_kpi` | 效果指标，如"家长续费率提升 5%" |
| `roi_estimate` | 收益预估：高 / 中 / 低 |

这部分人先填个大概，AI 帮你校准——比如你说"效果不错"，AI 追问"不错具体指续费率提升多少？"

### 1.5 来源追溯与系统内部字段

| 字段 | 说明 |
|------|------|
| `decision_id` | 来自哪条决议（可为空，非会议产生的 TDL 不关联） |
| `meeting_id` | 来自哪次会议（由 decision_id 派生） |
| `source` | 创建方式：meeting_minutes / dingtalk_msg / manual |

人不看、AI 和后台用：`tdl_id`（唯一编号）、`start_at`、`snooze_until`（暂缓恢复时间）、`calendar_event_id`（日历事件 ID）、`created_by`、`created_at`、`updated_at`、`audit_log`

## 四、一条 TDL 的一生（状态流转）

```
草稿 ──→ 进行中 ──→ 已完成
  │          │
  │          ├──→ 已逾期 ──→ 已暂缓 ──→ 进行中
  │          │       │
  │          │       └──→ 已升级
  │          │
  │          └──→ 已取消
  │
  └──→ 已取消（草稿 24 小时未确认自动取消）
```

| 状态 | 含义 |
|------|------|
| 草稿 | AI 生成，等确认 |
| 进行中 | 正在推进 |
| 已逾期 | 过了截止时间，系统自动标记 |
| 已暂缓 | 用户主动延迟，到时恢复 |
| 已升级 | 已通知上级 |
| 已完成 | 搞定 |
| 已取消 | 不做了 |

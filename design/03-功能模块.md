# 03 · 功能模块

> 与 02 的关系：每个模块操作的数据结构定义在 02。
> 与 04 的关系：每个模块的用户交互方式定义在 04。
> 与 06 的关系：每个模块的 MVP 范围定义在 06。

---

## 概述

基于 GTD 五步流程校准后的五模块架构：

```
记下来 ──→ 理清楚 ──→ 推得动 ──→ 看得见 ──→ 长出来
 意图捕获    结构化处理   进度推进    复盘可见    沉淀进化
   ✅          ✅          ✅         ✅         🔮
  MVP         MVP         MVP        MVP       第2阶段
```

---

## 模块一：意图捕获（记下来）

**解决什么**：月会上的决议、管理者正常的工作安排——这些东西不记下来、不拆成可追踪的任务就丢了。这个模块让系统被动接收，而不是人主动录入。

### 三个输入通道（按优先级排列）

**① 会议纪要转 TDL**（✅ MVP 首批）——权重最高的入口。

月会后用户贴入会议纪要文本（或上传录音转写），AI 自动拆出：决议→任务→责任人→截止时间→验收标准，批量生成 TDL 草稿。李珍主持的月度复盘会是信息密度最高、决策质量最高的场景，一次月会产出几十条有战略价值的 TDL。不做实时会议转写——只做纪要文本的后处理。

**② 日历自动写入**（✅ MVP）——TDL 创建后，系统自动在负责人的日历上生成一个日程事件。不是从日历读 TDL，是把 TDL 推到日历上。员工不需要自己建日历、不需要改习惯——打开日历就能看到待办。

**③ 日常对话录入**（🔮 第 2 阶段——基础 SOP 跑通后再做）——给机器人发一句话生成 TDL。这个通道的价值在于捕获碎片化待办，但在月会体系跑通之前，投入产出比不如会议纪要通道。

### 低风险自动创建，高风险草稿确认

不是每条消息都让人点确认——系统判断：

```
自动创建（发「已创建，可修改」卡片）：
  条件全部满足：截止时间明确 + 负责人明确 + 置信度≥85% + 不涉及他人

草稿确认（发「确认创建」卡片）：
  任一满足：缺截止时间、缺负责人、置信度低、涉及他人
```

## 模块二：结构化处理（理清楚）

**解决什么**：一句话进来，变成一条结构化的 TDL——有标签、有优先级、有下一步动作。

### AI 做四件事

**① 字段抽取**：从自然语言提取 title、due_at、owner、completion_criteria

**② 标签分类**：对照标签词典（02 文档 §2）打 business_line、function_domain、tags，置信度 < 70% 标「待确认」不硬塞

**③ 缺失检测**：缺截止时间？追问。缺完成标准？只在 AI 判断"标准不清楚"时才问——不每条都追问，不增加负担

**④ 下一步动作**：给一句简洁建议。子任务拆解限制：只有用户说了"方案/项目/规划/落地/推进"这类词才建议拆，不每条都拆成三步

💡 设计考量：MVP 不做 OKR 对齐检查——因为没有真实维护的 OKR 数据源，AI 会凭空编。

### 处理示例

```
用户：「下周三前审核斯坦星球暑期班方案，教研组配合」

AI 处理结果 → 飞书卡片：
┌──────────────────────────────────────┐
│ 📌 斯坦星球暑期班课程方案审核          │
│ ├─ 截止：2026-05-20 18:00            │
│ ├─ 业务线：斯坦星球 / 教务教学         │
│ ├─ 优先级：P1-重要不紧急              │
│ ├─ 等待：@教研组                      │
│ ├─ 完成标准：[待补充]                 │
│ └─ 下一步：联系教研组确定反馈截止时间   │
│                                      │
│ [✅ 确认] [📅 改时间] [👤 改负责人]   │
└──────────────────────────────────────┘
```

## 模块三：进度推进（推得动）

**解决什么**：任务不会自己推进。到时间提醒、逾期追问、卡住了升级——让系统做那个"不会忘、不怕烦"的角色。

### 提醒节奏（按班制路由）

系统根据每个人的班制类型决定提醒时间：

| 班制 | 适用岗位 | 每日提醒 | 周二特殊 |
|------|---------|---------|---------|
| 教师班 | 教师、教务、教务长、班主任、校长、培训师 | 08:30 | 正常 |
| 运营班 | 课程顾问、销售经理、市场/新媒体 | 08:30 | **延至 10:00**（周二仅 4.5h 工时，不给早上塞催收） |
| 标准班 | HR、人力主管、技术、财务、保洁 | 08:30 | 正常 |
| 上一休一 | 前台 | 有班日 08:30 | 休息日不推 |

```
每日固定时间 → 今天到期的 + 已经逾期的
                + 昨日完成确认：「昨天完成了 3 条 ✅」
每周一 09:00 → 本周到期的汇总
```

💡 班制数据来源于飞书薪资系统（参考 `assets/` 目录）。推广后系统从管理层名单自动读取班制，按上表路由提醒时间。

### 渐进式升级

不是突然升级——给缓冲、给出口，但不回复不让系统失效：

```
逾期 1 天 → 提醒本人
逾期 2 天 → 追问本人，带按钮：「[✅ 已完成] [📅 延期] [🆘 需协助]」
逾期 3 天 → 未回复 → 通知直属上级 + 写审计日志
逾期 7 天 → 继续升级（按职级链往上）
```

### 升级路由（按职级链 + 协作关系）

系统根据两条线决定通知谁：

**① 职级链（reports_to）**——逾期时沿汇报关系向上通知：

```
教师 T2/T3 → 教学主管（曹艳明/邢杨）→ 张蕾 → 李珍
班主任 → 教务主管（赵晓华）→ 张蕾 → 李珍
课程顾问 → 销售经理（时颖）→ 李珍
市场专员 → 市场经理（时颖）→ 李珍
HR专员 → 人力经理（张皓）→ 李珍
```

**② 协作链（blocked_by / waiting_for）**——当一条 TDL 因为等某人而阻塞时，同步通知那个人的上级：

```
例：邢杨的 TDL "W38续费冲刺" 等待赵晓华确认数据
  → 逾期后，不仅通知邢杨的上级（张蕾），也通知赵晓华
  → 如果赵晓华也逾期，她的上级（张蕾）收到"协作阻塞"通知
```

💡 协作链的意义：大部分逾期不是一个人的问题，是上下游断了。市场没给线索 → 销售做不了 Demo → 教学空着班。升级系统要知道"谁卡住了谁"，而不只是"谁没做完自己的事"。

### 管理人员链（完整）

| 人员 | 直属上级 | 协作关联人 |
|------|---------|-----------|
| 李珍 | — | 所有人 |
| 张蕾 | — | 赵晓华、曹艳明、邢杨、韩雪敏 |
| 荆少巍 | — | 孙娟、许梦冉 |
| 时颖 | 李珍 | 赵晓华（开班对接）、苗雅楠 |
| 苗雅楠 | 李珍 | 时颖、许梦冉 |
| 张皓 | 李珍 | 各部门用人需求方 |
| 许梦冉 | 李珍 | 苗雅楠 |
| 赵晓华 | 张蕾 | 曹艳明、邢杨、时颖（开班数量） |
| 曹艳明 | 张蕾 | 赵晓华（续费数据）、韩雪敏（新师输送） |
| 邢杨 | 张蕾 | 赵晓华、韩雪敏 |
| 韩雪敏 | 张蕾 | 曹艳明、邢杨（新师带班安排） |
| 李肖 | 苗雅楠 | — |
| 钟曼 | — | — |
| 孙娟 | 荆少巍 | 张皓、许梦冉（薪资数据） |

MVP 自用期：少巍自己就是"上级"，升级通知自己收。推广后按上表路由。

### 上级提前可见（非逾期）

推广期增加"上级简报卡片"——每周向上级推送下属本周的重点 TDL 清单。不是等逾期了才知道，而是提前知道团队在忙什么。交互细节见 04-交互方式 §上级简报卡片。

### 升级不等于惩罚

推送语气是关键——系统不说「逾期」，说「需要支持」：

| 场景 | 不好的说法 | 系统的说法 |
|------|-----------|-----------|
| 普通逾期 | "X 任务逾期 2 天未完成" | "X 任务可能需要关注" |
| 升级通知 | "X 已升级到你的上级" | "X 任务可能遇到了困难，已同步给 @上级" |
| 阻塞任务 | "X 被阻塞" | "X 任务建议协同推进" |

### 暂缓

用户说「明天再提醒」→ `snooze_until` = 明天 09:00，到时自动恢复。

## 模块四：复盘可见（看得见）

**解决什么**：一周过去了，做了什么、卡在哪、哪些事一直在等别人——不是凭感觉，是看数据。

### 周报（每周一 09:00 飞书推送）

只做事实统计，不做行为分析：

```
📊 TDL 周报（5.11-5.17）
├─ 本周新增：15 条
├─ 本周完成：12 条（80%）
├─ 逾期未清：3 条
├─ 延期处理：2 条
├─ 等待他人：4 条（@张三 2条，@李四 1条…）
├─ 阻塞中：1 条
├─ 超过 3 天未推进：
│   ├─ 励步招生方案终稿（5 天未动）
│   └─ 斯坦星球课程大纲修订（4 天未动）
├─ 下周到期：6 条
├─ 卡点提示：飞书工资相关 TDL 连续 2 周有阻塞
└─ 💡 下周建议：飞书工资线连续 2 周阻塞，建议下周单独约谈相关责任人
```

### 不做这些（第 2 阶段以后）

- 高效时段分析、精力曲线
- 拖延模式识别
- 团队排名
- Web 可视化仪表盘
- **老板督导航面板**（🔮 第 2 阶段）：本周必须盯到底的 N 件事、管理者风险名单、会议决议完成率、需要你拍板的问题

💡 MVP 周报已经做了五层模型第五层的前两步（记录+评估），加上"下周建议"补了第三步（调整）。和深度行为分析的区别：建议只从逾期/阻塞数据里推一条最明显的，不做多维交叉分析。
- 团队排名
- Web 可视化仪表盘

数据量不够时做行为分析是噪音。

## 模块五：沉淀进化（长出来）—— 第 2 阶段

**解决什么**：系统用久了，数据自己会说话——哪个业务线平均响应最快、什么类型任务容易卡、管理动作的投入产出怎么样。这不是第一优先级，但数据结构从第一天就要为它设计。

### 四个维度

| 维度 | 个人 | 部门/业务线 | 公司 |
|------|------|-----------|------|
| 经验 | 什么任务容易拖延 | 教研类比销售类平均慢多少 | 跨业务协作哪里最卡 |
| 成果 | 我完成的 TDL 累计影响 | 该业务线 Q3 完成率趋势 | 战略级 TDL 达成率 |
| 效能 | 高效时段与任务类型匹配 | 该部门人效趋势 | 整体响应速度变化 |
| 决策 | 做的决策与结果对照 | 方案调整前后效果对比 | 战略决策落地率 |

### MVP 做的事

模块五在 MVP 不开发功能，但数据库从第一天就预留字段：实际耗时、延期次数、关联业务成果。数据积累够了，第 2 阶段直接计算，不需要补数据。

---

## MVP 不做清单

以下明确排除在 MVP 之外：

- Web 可视化仪表盘——用飞书多维表格代替
- 长期行为记忆与画像——数据量不够时是噪音
- 钉钉接入——先跑通再说
- OKR 自动对齐检查——没有真实维护的 OKR 数据源
- 审批流/文档深度嵌入——平台工程，非 MVP 核心
- 团队公开排名——推广后才考虑
- 复杂日历联动——仅做单向写入（TDL→日历），不做双向同步
- 管理层全员推广——MVP 只有少巍自己用

🔮 第 2 阶段的四项（已记录不遗忘）：
- 日常对话录入（月会体系跑通后补）
- 老板督导航面板（本周必盯 N 件事 + 管理者风险名单）
- 管理者能力成长建档（沉淀进化模块的数据维度之一）
- TDL 模板结构校准 AI（三层目标：业务→管理→成长）
